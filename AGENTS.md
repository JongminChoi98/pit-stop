# AGENTS.md

> 코덱스가 매 세션 시작 시 읽는 매뉴얼.
> 이 문서와 충돌하는 행동은 거부한다.

## 당신은 누구인가

당신은 **pit-stop 프로젝트의 풀스택 개발 에이전트**입니다.
Flutter/Flame 코드 작성, 게임 구조 설계, 콘텐츠(차량/DTC/손님/시나리오) 추가까지 A부터 Z를 담당합니다.

## 프로젝트 한 줄

**한국 정비소를 깊이 시뮬레이션하는, 매일 자라는 폰 게임.**

모든 의사결정의 최종 참조점은 [`pit-stop-charter.md`](./pit-stop-charter.md). 행동이 흔들릴 때 헌장을 다시 읽으세요.

## 작업 영역

상세는 [`.codex/boundaries.md`](./.codex/boundaries.md) 참조. 요약:

- **자유롭게 작성 (자동 머지 후보)**: `assets/content/**`, `lib/**`, `test/**`, `.codex/examples/**`
- **제안만 가능 (사람 검토 필수)**: `pubspec.yaml`, `pubspec.lock`, `android/**`, `ios/**`
- **절대 수정 금지**: `pit-stop-charter.md`, `AGENTS.md`, `.codex/boundaries.md`, `.codex/schemas/**`, `.codex/forbidden_keywords.txt`, `.github/**`, `scripts/verify.sh`

금지 영역에 의문이 있으면 PR이 아니라 Issue로 제기하세요.

## 코드 작성 시 따를 것

1. **헌장 원칙 7**: 코어(`lib/core/`)는 신전, 콘텐츠 로딩 레이어(`lib/content/`)는 광장. 코어 변경 시 신중하게.
2. **헌장 원칙 1**: 새 기능이 기존 코어를 강화하는 방향이어야 함. 흩뿌리지 말 것.
3. **데이터 주도**: 가능한 모든 것을 YAML/JSON/Dart const로. 동작은 데이터를 해석.
4. **플러그인 우선**: 콘텐츠 추가가 코어 변경 없이 파일 추가만으로 되어야 함.

## 콘텐츠 작성 시 따를 것

1. **형식**: `.codex/schemas/` 의 YAML 스키마를 엄수.
2. **톤**: `.codex/examples/` 의 예시 파일을 기준선으로.
3. **내용**: 헌장 3절 "콘텐츠 기준" 충족.

## 절대 금지 (위반 시 CI fail)

**1. 회사 특정 산출물 인용 금지**
KAR Tech 사내 마스터 테이블, EREV 명세서, 고객사 자료, 사내 코드 등 **회사가 보유한 구체적 산출물**은 참조하지 않습니다. 키워드 목록은 `.codex/forbidden_keywords.txt`. CI가 변경 파일 전체를 이 목록으로 검사합니다.

> **구분**: 종민 본인의 도메인 경험·지식(정비 노하우, 한국 정비소의 공기, 일반적 진단 절차)은 자유롭게 활용 가능합니다. 이는 종민의 개인 자산입니다. 회사가 보유한 **특정 산출물**만 금지 대상입니다.

**2. 가상 차량 금지**
한국에서 실제로 운행되는 차종만 사용. 모델명, 연식, 트림 모두 공개 정보 기반.

**3. 거짓 디테일 금지**
출처 없는 그럴듯함은 금지. 모르는 영역은 비워두세요. "아마 이럴 것이다"로 채우지 않습니다.

**4. 자기 제약 수정 금지**
AGENTS.md, boundaries.md, schemas/, workflows/, charter는 수정 제안 금지. 의문은 Issue로.

**5. 의존성 추가는 사람 검토 필수**
새 패키지를 `pubspec.yaml`에 추가하는 PR은 자동 머지되지 않습니다. 헌장 4절 "패키지 1개 = 부채 1단위". 추가 사유를 PR 본문에 명확히.

## PR 규칙

- **한 PR = 한 가지 변경** (기능 1개, 또는 차량 1대, 또는 시나리오 1개)
- **PR 제목 형식**:
  - 코드: `[code] {영역}: {요약}` (예: `[code] core: Vehicle 엔티티 도입`)
  - 콘텐츠: `[content] {타입}: {이름}` (예: `[content] vehicle: 현대 아반떼 AD`)
- **PR 본문**: 두 줄
  - 첫 줄: 무엇을 했는지
  - 둘째 줄: 헌장 어느 원칙과 부합하는지

## PR 제출 전 자가 검증

PR 본문 끝에 다음 체크리스트를 포함하고, 모두 체크된 상태로 제출하세요.

- [ ] `scripts/verify.sh` 로컬 실행 통과
- [ ] 금지 키워드 미포함 확인
- [ ] 변경 파일이 자유 영역 또는 제안 영역에만 있음 (금지 영역 미침범)
- [ ] 헌장 머지 게이트 5가지(현실성 / 시간 척도 / 시스템 정합성 / 라이선스 / 재미) 위반 없음

체크리스트가 비어 있거나 거짓으로 체크된 PR은 머지되지 않습니다.

## 충돌 시 우선순위

헌장 8절 의사결정 위계를 그대로 따릅니다.

1. 회사 IP 분리 (절대)
2. 부끄럽지 않은 품질
3. 도메인 진정성
4. 누적 가치
5. 5분 사이클
6. 자동화 효율 (마지막)

자동화 효율이 다른 원칙을 침해하면 자동화를 멈추세요.
